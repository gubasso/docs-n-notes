# 00 — Overview

A flake pins the toolchain (language runtimes + system tools) and gives every checkout a
reproducible dev shell. [direnv](https://github.com/nix-community/nix-direnv) enters that shell
automatically on `cd`. Dependency resolution stays with the language's own package manager (Poetry,
npm, cargo, …).

- [Nix flakes](https://nix.dev/concepts/flakes.html)
- [nix.dev — declarative shell](https://nix.dev/tutorials/first-steps/declarative-shell.html)
- [nix-community/nix-direnv](https://github.com/nix-community/nix-direnv)

## The three files (per project)

Every Nix-managed project commits exactly three files:

- **`flake.nix`** — declares `devShells.default` (the toolchain). Committed.
- **`flake.lock`** — pins every input to an exact revision. Committed; generated by `nix`, **never**
  hand-written.
- **`.envrc`** — `use flake` (plus, for Python, a Poetry-venv layer — see
  [02-per-project-devshell](02-per-project-devshell.md)), so direnv auto-enters the shell on `cd`.

Also add `.direnv/` and `/result` to `.gitignore`.

## Mental model

```
cd ~/Projects/foo          # .envrc present + trusted
      │
      ▼  direnv hook fires
use flake  ─────────────►  nix builds/loads devShells.default  ─────►  toolchain on PATH
      │                     (cached by nix-direnv after first eval)
      ▼
cargo / poetry / pnpm ...  # deps resolved by the language manager, inside the shell
```

Leave the shell (`cd ..`) and direnv reverses the PATH changes. Nothing is installed globally per
project; the toolchain lives in the Nix store and is garbage-collected only when no gcroot
references it (nix-direnv keeps one under `.direnv/`).

## Why not just `nix profile add` everything?

`nix profile` is an imperative global profile — good for **user CLIs** you always want.
Per-**project** toolchains belong in that project's flake so each repo pins its own node/python/rust
version reproducibly, independent of what's globally installed.
