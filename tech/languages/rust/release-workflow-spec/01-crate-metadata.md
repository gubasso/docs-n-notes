# 01 — Crate metadata

crates.io validates `[package]` metadata at publish time. Some fields are **required** (the publish
is rejected without them), some produce **warnings**, and some are optional but improve discovery.
Get this right before you ever create a token — a dry run surfaces every problem without needing
auth.

## Required (publish is rejected without these)

- `name` — must be unique on crates.io and unclaimed. Names are first-come and permanent once
  published.
- `version` — a valid [SemVer](07-semver-yank-rollback.md) version.
- `description` — a short, one-sentence summary. **crates.io hard-rejects a publish that has no
  `description`.** This is the single most common first-publish blocker.
- `license` **or** `license-file` — an [SPDX license expression](https://spdx.org/licenses/) (e.g.
  `MIT`, `Apache-2.0`, `MIT OR Apache-2.0`) or a path to a license file. One of the two is required.

## Recommended (warns, or just good practice)

- `repository` — source URL. crates.io warns when it is missing and it drives the "Repository" link
  on the crate page.
- `readme` — path to the README rendered on the crate page. Cargo auto-detects `README.md` if unset;
  set it explicitly to be unambiguous.
- `keywords` — up to **5**, each **≤20 chars**, alphanumeric plus `-`/`_`. Used by crates.io search.
- `categories` — must match the canonical
  [category slugs](https://github.com/rust-lang/crates.io/blob/main/src/boot/categories.toml)
  exactly, or the publish is rejected. Common ones: `command-line-utilities`, `development-tools`,
  `virtualization`.
- `rust-version` — the MSRV; lets Cargo error early instead of failing mid-build.
- `homepage`, `documentation` — extra links on the crate page (`documentation` defaults to docs.rs).
- `exclude` / `include` — trim what ships in the `.crate` tarball (test fixtures, large assets, CI
  scaffolding). `cargo package --list` shows exactly what will be uploaded.

## Example

```toml
[package]
name = "my-crate"
version = "0.1.0"
edition = "2021"
description = "One-sentence summary of what the crate does."
license = "MIT OR Apache-2.0"
repository = "https://github.com/<owner>/<repo>"
readme = "README.md"
keywords = ["cli", "example", "tooling"]
categories = ["command-line-utilities", "development-tools"]
rust-version = "1.74"
```

Dual licensing under `MIT OR Apache-2.0` is the Rust-ecosystem default. If you use `license-file`
instead, point it at the file:

```toml
[package]
license-file = "LICENSE"
```

## LICENSE file

Ship an actual license file in the repo even when you set the `license` SPDX field — it is what
readers and downstream tooling look for. With `MIT OR
Apache-2.0`, include both `LICENSE-MIT` and
`LICENSE-APACHE`.

## Package contents — keep the tarball lean

By default Cargo packages your **entire working tree** (minus `target/` and gitignored files), so a
crate with an extensive `docs/` tree, CI config, and dev tooling can ship dozens of files that have
nothing to do with building it. The Cargo Book's publishing guide calls this out directly: check the
`.crate` size and strip "large assets that are not required to build your package, such as test
data, **website documentation**, or code generation".

**What belongs in the tarball:**

- `src/`, `Cargo.toml`, `Cargo.lock` (auto-included), `build.rs` if present.
- `README.md` (the `readme` target) and `LICENSE` — for the crates.io page and legal compliance.
- `CHANGELOG.md` if you keep one.

**What does not belong** (none are build inputs):

- Project docs — specs, ADRs, guides, a `docs/` site. These live in the git repo (that is what
  `repository` is for) or on docs.rs / GitHub Pages.
- CI and dev tooling — `.github/`, `.pre-commit-config.yaml`, `deny.toml`, `release-plz.toml`,
  `dist-workspace.toml`, `justfile`, `flake.nix`, `.envrc`, editor/linter configs, `scripts/`.

**API docs vs project docs.** The ecosystem separates two things: **API docs** generated by rustdoc
from `///` comments in `src/` (auto-built and hosted on [docs.rs](https://docs.rs/)) travel _inside
the source_ and need no `docs/` tree; **project docs** (prose, ADRs, architecture) live in the repo
and are reached via the crates.io "Repository" link. Shipping a `docs/` tree in the `.crate` gives
readers nothing they can't already get from the repo — and relative links in the README resolve
against the repo on the crates.io page regardless.

### `exclude` vs `include`

Two mutually-exclusive manifest keys control this (`include` overrides `exclude`):

- **`exclude`** (denylist) — package everything _except_ the listed patterns. Lower risk of dropping
  a required file, since new `src/` files are included by default; you must remember to add each new
  junk pattern.
- **`include`** (allowlist) — package _only_ the listed patterns. The strongest guarantee that
  nothing extraneous leaks, but you must keep it complete or you silently drop needed files.

**Footgun:** Cargo _always_ includes `Cargo.toml`, the minimized `Cargo.lock`, and a `license-file`
— but **not** a plain `LICENSE` file, and **not** the file named by `readme`. So if you use
`license = "MIT"` (an SPDX expression) rather than `license-file`, an `include` allowlist **must**
list `README.md` and `LICENSE` explicitly or it will drop them. For this reason, `exclude` is the
safer default for most crates; reach for `include` only when you want an airtight allowlist and are
willing to maintain it.

```toml
[package]
# Denylist — robust against dropping future source files.
exclude = [
    "/docs",
    "/.github",
    "/scripts",
    "/deny.toml",
    "/release-plz.toml",
    "/dist-workspace.toml",
    "/justfile",
    "/flake.nix",
    "/flake.lock",
    "/.pre-commit-config.yaml",
]
```

Leading `/` anchors a pattern to the package root. Re-review the list whenever you add a new
top-level config artifact, so it does not silently start shipping.

### Size limit

crates.io enforces a **hard 10 MB limit** on the `.crate` file. Most crates are far under it, so
trimming is usually about hygiene and courtesy rather than the ceiling: every
`cargo build`/`cargo install` that touches your crate downloads and unpacks whatever you ship. For a
**binary crate** the point is sharper — it is only ever `cargo install`ed, never depended on, so _no
consumer reads any file in the tarball_; docs and tooling are pure dead weight to every installer.

## Verify before publishing (no token required)

```bash
cargo publish --dry-run   # builds + validates metadata, uploads nothing
cargo package --list      # exact file list that would ship
```

The dry run is the acceptance gate for this chapter: it fails loudly on a missing `description`, an
invalid `categories` slug, or a build error, and it needs no crates.io auth. See
[06 — Helper scripts](06-helper-scripts.md) for wrapping these in a `publish-dry` script.
