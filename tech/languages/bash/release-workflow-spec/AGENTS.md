# AGENTS

## Scope

Bash-specific release-and-distribution spec: the **bash binding** of the general
`tech/programming/release-workflow/` principles. Targets a single-maintainer, pure-Bash CLI (no
compiled artifacts, "noarch"). Covers versioning/VERSION SoT, conventional commits + git-cliff
changelog, the GNU-conventions Makefile, the tag-triggered CI release job, and the end-user
distribution channels (`install.sh`, AUR, OBS for `.rpm`/`.deb`).

## Key Points

### Divergence from the registry-publish model

- Bash has **no central package registry**. A "release" is a signed `v*` git tag + a GitHub/GitLab
  Release + distribution channels. **Tagging is publishing** — nothing to
  `cargo publish`/`npm publish`.
- The signed `v*` tag is the single source of truth; every channel keys off it.

### Versioning

- `VERSION` file at repo root, one line, SemVer. Expected by GNU standards, OBS `set_version`,
  `nfpm`, RPM `%autosetup`.
- `version` subcommand: either ship `VERSION` into `$(datadir)/<tool>/VERSION`, or substitute a
  `__TOOL_VERSION__` placeholder at install time. Dev checkouts fall back to
  `git describe --tags --dirty --always`.

### Changelog

- Conventional Commits enforced by a `commitlint` pre-commit hook.
- `CHANGELOG.md` generated by `git-cliff` (`cliff.toml`, Keep-a-Changelog preset). Chosen over
  release-please (which is GitHub/Node-centric and PR-based); git-cliff runs anywhere and you tag
  manually.

### Makefile (GNU conventions)

- Highest-leverage change. Define `PREFIX`/`DESTDIR`/`bindir`/`libdir`/`datadir`/`sysconfdir`; every
  install path is `$(DESTDIR)$(prefix-var)` — the contract every packaging tool assumes.
- `user-install` sets `PREFIX=$HOME/.local`. `dist` uses `git archive` + `sha256sum` for a
  reproducible tarball. Provide a symmetric `uninstall`.

### CI release workflow

- `.github/workflows/release.yml`, trigger `on push tags ['v*']`. Permissions: `contents: write`,
  `id-token: write`, `attestations: write`.
- Jobs: `test` → `release` (`make dist` → `orhun/git-cliff-action` notes →
  `actions/attest-build-provenance` (SLSA L3, keyless via Sigstore/GitHub OIDC) →
  `gh release create` → trigger OBS service run) → `promote` (fast-forward `master` onto the tag,
  ancestry-checked against `develop`, `--ff-only`; keeps `master` CI-only).
- OBS trigger: `POST api.opensuse.org/trigger/runservice`, header literal `Authorization: Token`
  (never `Bearer`), `--fail-with-body`. Needs `OBS_TOKEN` repo secret.

### Distribution channels (priority order)

- `install.sh` (`curl | bash`): discover latest tag via Releases API → download tarball + `.sha256`
  → **verify checksum before extracting** → `make install` (default `PREFIX=$HOME/.local`).
- AUR: two packages — `<tool>` (pins latest tag) and `<tool>-git` (VCS variant tracking `master`).
  `package()` is one line thanks to the DESTDIR/PREFIX Makefile. PKGBUILDs live in `packaging/aur/`,
  pushed via `git subtree`.
- OBS: builds `.rpm`+`.deb` from one `.spec` + flat `debian.*` set (`debtransform`) and **hosts
  signed repos** for Tumbleweed/Leap/Fedora/Debian/Ubuntu. `_service` uses `obs_scm`
  (`mode="manual"`, `revision master`, version from `@PARENT_TAG@`). `BuildArch: noarch`. One-time
  setup: sub-project, `_meta` repos, package container, scoped `runservice` token, sanity-check in a
  `test-` project before renaming (renaming a published project breaks users' repo URLs).

### Branch model & alternatives

- Working branches are `develop`/`master`; the tag is cut on `develop` and **CI fast-forwards
  `master` onto it** (a `promote` job, ancestry-checked, `--ff-only`) — `master` is written only by
  CI.
- Release ritual: `git-cliff --bump` → write `VERSION` → commit → signed `git tag -s` →
  `git push --follow-tags`; CI builds the release and promotes `master`.
- Alternatives noted (when to pick): release-please, GoReleaser, nfpm (attach `.rpm`/`.deb` to the
  Release with no hosted repo), Fedora COPR, self-hosted repos, OBS-Arch (additive to AUR, never a
  substitute).

## Maintenance Notes

- Regenerate when any chapter changes, when repo names bump (Leap/Fedora versions), or when the
  Actions/OBS action versions shift.
- Branch naming is standardized to `develop`/`master` throughout; if a chapter reintroduces
  `main`/`devel`, fix it and re-sync.
- `master` is **CI-only**: the tag is cut on `develop` and a `promote` job fast-forwards `master`
  onto it (`03-ci-release-workflow.md`), matching the general model. Do not reintroduce "tag cut on
  `master`".
- `runbook.md` is the per-new-project setup guide (parallels the rust shelf's `runbook.md`); the
  recurring release ritual stays in `06-release-ritual-and-alternatives.md`.
- This is the bash binding of `tech/programming/release-workflow/` (authored in parallel); the
  back-link target may not exist yet.
